## 🟢 Foundational Scenarios

**Scenario 1. Your Canvas App works fine in testing with 200 rows in a SharePoint list, but in production (12,000 items) users report filters return incomplete/wrong results. What's happening and how do you fix it?**
This is a delegation issue — SharePoint's non-delegable query limit truncates the working set at roughly 500–2,000 records before your `Filter`/`Search` formula ever processes the rest. Fix by rewriting formulas to use only delegable operators for the connector, indexing the filtered columns in SharePoint, considering migrating the list to Dataverse (much higher delegation limits and richer querying), or pre-filtering server-side via a scheduled Power Automate sync into a smaller working table.

**Scenario 2. Finance wants an approval process where an invoice over $10,000 needs both Manager and Finance Director sign-off (in sequence), but under $10,000 only needs Manager approval — built on a SharePoint list with a Power Automate flow. Design the flow.**
Trigger on item create/modify; add a Condition checking `Amount >= 10000`. In the "Yes" branch, run a sequential "Start and wait for an approval" to the Manager, then (only if approved) a second approval to the Finance Director, updating the SharePoint status column after each stage. In the "No" branch, run just the Manager approval. Use "Configure run after" so a rejection short-circuits without proceeding further, and log each decision (approver, comments, timestamp) for auditability.

**Scenario 3. Users report a Model-Driven App form is slow to load, especially a specific tab with a sub-grid. How do you troubleshoot?**
Use browser DevTools' Network tab (filtering `api/data` calls) alongside Dataverse's form diagnostics to check whether a synchronous plugin on Retrieve, an overly broad/unfiltered sub-grid view, or an inefficient synchronous JavaScript `OnLoad` script is the culprit; fixes typically include adding server-side filtering to the sub-grid, converting synchronous form-load calls to async, and reviewing plugin registration steps for unnecessary synchronous, unfiltered triggers.

**Scenario 4. A tenant-wide SPFx Application Customizer starts causing multiple sites to load slowly after an update. How do you diagnose and roll back safely?**
Diagnose with DevTools Performance/Network tabs or the Page Diagnostics Tool to identify whether the new version added a slow synchronous call in `onInit()`; redeploy the previous known-good `.sppkg` version via "Replace" in the App Catalog (same solution ID) as an immediate rollback, then fix the root cause and re-validate in a pilot site before redeploying tenant-wide.

**Scenario 5. A Power Automate flow that sends a Teams notification on every SharePoint list update is firing hundreds of times a day and users have muted the channel. How do you fix it?**
Add a Trigger Condition so the flow only fires when a specific meaningful field actually changes (e.g., `Status` transitioning to "Approved"), rather than on any modification including unrelated columns; if even filtered real-time notifications are too frequent, consider a scheduled digest flow summarizing changes instead of real-time-per-item alerts.

## 🟠 Advanced Scenarios

**Scenario 6. You need to migrate a legacy InfoPath form plus a SharePoint 2013 workflow (multi-stage HR onboarding) to the modern platform. What's your target architecture?**
Replace InfoPath with a Power Apps Canvas App (embedded via the "customize form with Power Apps" option) or a Model-Driven App if migrating the data to Dataverse for richer relational modeling of onboarding tasks; replace the 2013 workflow with a solution-aware Power Automate Cloud Flow handling the multi-stage approvals/notifications; if the process needs a guided step-by-step experience for HR staff, add a Dataverse Business Process Flow on top. Package everything into a Dataverse Solution for proper Dev→Test→Prod ALM.

**Scenario 7. A customer-facing Power Pages site accidentally exposed internal-only Dataverse records to anonymous visitors. What went wrong and how do you prevent recurrence?**
A Table Permission was almost certainly scoped too broadly (e.g., "Global" instead of "Contact"/"Account-owned") for an overly permissive Web Role; correct the permission immediately and audit all other tables for the same pattern. Prevent recurrence with a least-privilege-by-default posture, peer review of permission changes before production, and periodic security audits of the whole permission model.

**Scenario 8. Leadership wants a single Power BI report combining near-real-time sales data (changes every few minutes) with a large historical warehouse (5 years, rarely changes). How do you architect the dataset?**
Build a Composite model: large historical tables in Import mode (fast, nightly-refreshed), the near-real-time sales table in DirectQuery (or Import with frequent incremental refresh); use Dual storage mode on shared dimensions so they work efficiently with either mode, and apply Incremental Refresh on the historical fact table so nightly refreshes only touch recent partitions.

**Scenario 9. A Power Automate Desktop unattended flow that used to work reliably starts failing intermittently after a target application's UI was updated. How do you make the automation more resilient long-term?**
Short-term, update the broken selectors. Long-term, reduce reliance on brittle absolute selectors (favor UIA-based relative/anchor selectors), add "Wait for element" instead of fixed delays, wrap steps in "On Block Error" with retry logic and screenshot-on-failure diagnostics, and evaluate whether the target system has since exposed an API that could replace this fragile UI dependency.

**Scenario 10. Your organization wants a Copilot Studio bot to answer HR policy questions using existing SharePoint HR documents, but must ensure an employee never sees a policy from a department they don't have SharePoint access to.**
Configure the SharePoint HR library as a Generative AI Knowledge Source, and ensure the agent runs with the *end user's* identity context (not a service account) so Microsoft Search's security trimming applies at retrieval time — then validate with multiple test accounts holding different SharePoint permissions before broadly publishing the bot.

**Scenario 11. A SPFx web part needs to call three different SharePoint/Graph endpoints, and the current implementation calls them sequentially inside `componentDidMount`, causing a 3–4 second load delay. How do you optimize?**
Fire all three independent calls concurrently with `Promise.all([...])` instead of sequential `await`s, kick off fetching as early as possible (constructor/`onInit()`), add loading skeletons for perceived performance, and consider PnPjs caching (`usingCaching()`) for any call returning relatively static data.

**Scenario 12. A Power Automate cloud flow that processes uploaded invoices via AI Builder Form Processing occasionally extracts wrong totals for a new vendor's invoice layout. How do you fix and prevent this?**
The custom model likely wasn't trained on this vendor's layout — add labeled samples of the new layout and retrain/republish (Form Processing models are layout-specific). For prevention, add a confidence-threshold check routing low-confidence extractions to a human-review step instead of blindly trusting them, and establish a periodic retraining process as new vendor layouts are onboarded.

## 🔴 Expert / Architect-Level Scenarios

**Scenario 13. You're asked to design an ALM strategy for a Power Platform solution spanning a Model-Driven App, three Cloud Flows, a Canvas App, and a Power BI report — shared across Dev, Test, and Production environments.**
Build everything inside a single (or logically grouped) Dataverse Solution in Dev, using Environment Variables for environment-specific config and Connection References for every connector so no personal/hardcoded connections are embedded; export as a Managed Solution and promote via Power Platform Pipelines (or Azure DevOps/GitHub Actions with Power Platform Build Tools) into Test, then Production. For the Power BI report, use Deployment Pipelines in parallel, rebinding the dataset's data source per stage. Maintain the unpacked solution (`pac solution unpack`) in a Git repo for version control and rollback capability rather than relying solely on maker-portal export history.

**Scenario 14. Design an end-to-end citizen-services experience: an external applicant submits a permit application via Power Pages, an AI Builder model extracts data from an uploaded supporting document, a Copilot Studio bot answers applicant questions about status, and a Power BI dashboard tracks processing SLAs for internal staff — all built on a shared Dataverse backend.**
Model the core "Application" table in Dataverse with appropriate Table Permissions so the Power Pages site only exposes each applicant's own records (Contact-scoped); the uploaded document triggers a Power Automate flow calling a custom AI Builder Form Processing model to extract structured data into the Application record, routing low-confidence extractions to an internal review queue; the Copilot Studio bot is grounded on a curated internal knowledge base (not raw case data) for general questions, with a separate Action calling a flow to securely look up the specific applicant's own case status (validated against their authenticated identity, never trusting a bare case number typed into chat as sufficient authorization); and the internal Power BI dashboard connects live (or via scheduled refresh) to the same Dataverse tables, using Row-Level Security so different internal teams see only their jurisdiction's applications. All components live in one governed Dataverse Solution for unified ALM.

**Scenario 15. A merger requires consolidating two organizations' SharePoint tenants, Power Platform environments, and a mix of Canvas Apps, Model-Driven Apps, and Power BI reports built independently by each company. How do you approach this at a program level?**
Start with an inventory/discovery phase (using the Center of Excellence Starter Kit and SharePoint/Power Platform admin center reporting) to catalog what exists in both tenants — apps, flows, datasets, connectors used, and their actual business criticality/usage, since a merger is the right moment to retire redundant or low-value assets rather than migrating everything by default; establish a target governance model (naming conventions, environment strategy, DLP policies) before migration begins rather than retrofitting it after; migrate SharePoint content using a tenant-to-tenant migration tool with a pilot site collection first to validate permission/identity remapping; and for overlapping capabilities (e.g., both companies have their own expense-approval Power Automate flow), make a deliberate consolidation decision (choose one, rebuild a merged version, or run both temporarily during transition) rather than allowing duplicate, diverging systems to persist indefinitely post-merger.

**Scenario 16. A cross-cutting production incident: a Power Automate flow triggered by a Dataverse record update calls a Copilot Studio Action, which in turn calls another flow that writes back to the same Dataverse table, and the system enters a runaway trigger loop consuming massive API quota before anyone notices. How do you both resolve the immediate incident and redesign to prevent recurrence?**
Immediately, disable/pause the implicated flows to stop the loop and prevent further quota consumption or throttling impact on other tenant workloads, then review System Jobs/flow run history to confirm the exact chain of triggering. For redesign, add trigger conditions to each flow checking whether the *specific field* relevant to that flow's own logic actually changed (not "any update"), introduce a "processed" flag or version-check pattern so a flow doesn't re-act to a change it itself just made, and explicitly map out which flows/agents watch which tables and fields during design/architecture review — treating this kind of cross-component dependency mapping as mandatory before any new automation touches a table that other flows or Copilot Actions already write to.

**Scenario 17. Leadership wants a unified employee "digital front door" — a single Copilot Studio agent in Teams that can answer HR questions (grounded on SharePoint), submit IT tickets (via Power Automate into a Dataverse-based ticketing Model-Driven App), and show each employee their personal PTO balance via a live Power BI-style visual — all while respecting each system's existing security model. How do you architect this?**
Keep each capability's underlying security model authoritative and unchanged (SharePoint permissions still govern HR document access, Dataverse security roles still govern ticketing data, and the PTO balance lookup respects whatever system of record already secures HR/payroll data) and have the Copilot Studio agent act strictly as an orchestration/UX layer on top, never as a bypass: ground HR Q&A on SharePoint using the end user's own identity for search security trimming, use an Action calling a Power Automate flow (with proper Dataverse security context) to create IT tickets, and use another Action to securely retrieve just that authenticated employee's own PTO balance from the source system rather than exposing a broader dataset the LLM could inadvertently over-share; test with multiple accounts of varying permission levels across all three integrated systems before rollout, since a single unified conversational front end makes it easy to overlook that three entirely different security models must each still be correctly enforced underneath.

**Scenario 18. Your organization is being audited for regulatory compliance (e.g., SOX or similar), and auditors want to trace exactly who approved a specific high-value transaction, when, and confirm the approval logic couldn't have been bypassed — spanning a Power Apps form, a Power Automate approval flow, and Dataverse records.**
Ensure Dataverse's built-in Auditing is enabled on the relevant tables (capturing field-level change history, including who/when), confirm the approval logic is enforced server-side (e.g., a status field that can only transition via the flow's approval outcome, ideally locked from direct user edit through field-level security or a plugin, rather than being an easily-editable field a user could manually change to "Approved" bypassing the flow entirely), retain full Power Automate run history/logs for the retention period auditors require, and be able to demonstrate — through the combination of Dataverse audit trail, flow run history, and the approval action's own logged approver/timestamp/comments — an unbroken, tamper-evident chain of evidence for that specific transaction rather than relying on any single system's logs in isolation.
